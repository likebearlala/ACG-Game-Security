# 劇情生成技術說明

本專案的劇情預設由前端 `game.js` 內建的敘事規則、隨機資料池、事件模板、數值系統與結局條件組合出來；玩家也可以選擇啟用 LLM API 生成模式，讓瀏覽器用玩家輸入的 API Key 呼叫 OpenAI-compatible Chat Completions。API Key 只保存在本頁記憶體，不寫入存檔。

## 核心概念

遊戲採用「規則式互動敘事」：

- 開局設定決定角色身份、玩家性別、攻略對象性別、目前關係、危險對象類型、故事風格與情感濃度。
- 系統從名字池、特質池、頭像池中隨機抽出角色資料。
- 玩家每次選擇都會觸發一個事件模板。
- 事件模板會改變主要攻略角色、危險對象與玩家自己的數值。
- 數值累積後會推進關係階段，並影響後續可選行動與結局。

所以每一輪不是完全隨機，也不是固定一本道，而是「固定劇情素材 + 隨機抽取 + 數值分歧」的混合生成。

## 開局資料來源

`setupSteps` 定義開局流程，目前包含七步：

1. 身份
2. 自己的性別
3. 目前關係
4. 攻略對象性別
5. 攻略角色
6. 故事風格
7. 情感濃度

玩家每一步的選擇會存進 `setupData`。按完最後一步後，`finishSetup()` 會把這些選項整理成正式的 `game` 狀態。

## 角色生成

角色生成主要由 `pools` 和 `finishSetup()` 負責。

`pools` 裡放了：

- 男性名字池
- 女性名字池
- 中性名字池
- 玩家性別隨機池
- 攻略對象性別隨機池
- 角色特質池
- 職業池

當玩家選擇攻略對象性別後，`namesForTargetGender()` 會決定名字抽取範圍：

- 選男性：主要攻略角色與危險對象都從男性名字池抽。
- 選女性：兩者都從女性名字池抽。
- 選男女皆有：主要攻略角色與危險對象會分別來自男女名字池。
- 選不限制或自訂：從全部名字池抽。

`finishSetup()` 也會避免主要攻略角色和危險對象抽到同一個名字。

## 頭像生成

頭像由 `avatarPool`、`randomAvatars()` 和 `avatarFor()` 管理。

開局時 `finishSetup()` 會呼叫 `randomAvatars()`，從可用頭像中隨機分配：

- 玩家頭像
- 主要攻略角色頭像
- 危險對象頭像

同一輪遊戲中，這組頭像會存進 `game.avatars`，所以側欄、數值變動彈窗和結局頁會保持一致。讀取舊存檔時，如果沒有 `avatars` 欄位，系統會自動補一組隨機頭像。

## 劇情事件生成

主要事件由 `buildScene(choice)` 生成。玩家選擇會帶有一個 `type`，例如：

- `stable`：主要攻略角色路線
- `danger`：危險對象路線
- `self`：玩家獨處或迴避
- `social`：IG / 社群動態
- `lie`：欺瞞、刪訊息、定位、報復等危險行動

`buildScene()` 內部有 `scenePools`，每個類型都有多個事件模板。事件模板包含：

- `id`：避免同一輪一直重複抽到同一事件
- `title`：場景標題
- `delta`：數值變動
- `paragraphs`：實際劇情段落

每次玩家行動時，系統會從對應事件池中抽一個尚未用過的事件。如果該類事件都用過了，仍可重複抽取，但會追加一段「重複行為留下痕跡」的文字。

## 數值系統

遊戲有三組數值：

- 主要攻略角色：愛意、信任、親密、不安、距離
- 危險對象：心動、吸引、依賴、主動、在意
- 玩家：壓力、猶豫、界線感

事件的 `delta` 會透過 `applyDelta()` 套用到這些數值上。所有數值會被限制在 0 到 100 之間。

數值的用途包括：

- 決定關係階段
- 觸發額外壓力
- 改變下一輪選項
- 判定結局

## 關係階段推進

`advanceStage()` 會根據數值決定目前階段，例如：

- 危險對象心動或吸引升高
- 主要攻略角色不安升高
- 玩家壓力升高
- 信任降低或距離升高

階段文字存在 `stages` 中，會顯示在畫面上。階段越高，代表關係越接近爆發或結局。

## 每週事件

玩家每週有 5 個行動點。行動點用完後會進入下一週。

`weeklyEvent()` 會在新的一週開始時插入週事件。它會根據當前數值選擇不同情境，例如：

- 最後幾週倒數
- 主要攻略角色質問
- 危險對象要求更明確的關係
- 普通週事件

這讓劇情不只依賴玩家當下選擇，也會受到整體關係壓力影響。

## 結局判定

結局由 `resolveEnding(reason)` 判定。它會讀取目前三組數值，判斷故事走向。

目前包含的主要結局類型：

- 主動坦白
- 悄然抽身
- 危險對象失控
- 徹底翻車
- 雙線崩盤

例如：

- 玩家猶豫很高，但主要攻略角色仍有足夠愛意與信任，可能進入「主動坦白」。
- 危險對象主動或在意過高，可能進入「危險對象失控」。
- 主要攻略角色不安過高、信任過低，可能進入「徹底翻車」。
- 到第 12 週仍沒有明確處理，可能進入期限結局。

## 為什麼看起來像生成式劇情

即使未啟用本地 AI，劇情也會因下列因素產生變化：

- 開局選項不同
- 角色名字與特質隨機
- 頭像隨機
- 事件抽取順序不同
- 玩家選擇路線不同
- 數值累積不同
- 週事件與結局條件不同

這些變因組合後，會讓每一輪的劇情順序、角色感覺與結局走向不同。

## LLM API 生成模式

目前專案會在真正開始遊戲前要求玩家設定 LLM API。玩家完成七步開局設定後，畫面會停在 LLM API Key 設定頁，必須選擇其中一條路徑：填入 API Key 啟用生成劇情，或明確選擇陽春版。

- API Key
- OpenAI-compatible Chat Completions endpoint
- Model 名稱

預設 endpoint 是：

```text
https://api.openai.com/v1/chat/completions
```

API Key 只存在瀏覽器記憶體中，不會寫入 localStorage，也不會跟遊戲存檔一起保存。玩家也可以選擇「陽春版」，完全不使用 LLM API。

這個模式的特點：

- 不做瀏覽器本地模型載入。
- 不做 GPU 能力偵測。
- 由玩家自己的 API Key 承擔 LLM 呼叫。
- 若 API 呼叫失敗或回覆格式不合格，該段會 fallback 到陽春版規則劇情。

## LLM API 如何接入劇情

原本的規則式事件仍由 `buildScene()` 和 `weeklyEvent()` 產生基礎場景，包含：

- 場景標題
- 原始段落
- 數值變動
- 行動類型

輸入 API Key 並啟用 LLM 後，流程變成：

1. 玩家做出選擇。
2. `buildScene()` 先產生一個基礎事件。
3. `llmScenePrompt()` 將目前 `game` 狀態、角色資料、數值、最近選擇與原始段落整理成 prompt。
4. `generateLLMScene()` 用玩家輸入的 API Key 呼叫 OpenAI-compatible Chat Completions。
5. 模型回傳完整視覺小說節點：標題、小說段落、下一步選項、數值變動，以及可選的結局。
6. 系統會檢查 JSON 格式、限制數值 key 與變動幅度；LLM 成功時直接使用它生成的選項，失敗時才 fallback 到陽春版規則選項。

這樣做的原因是：LLM 負責把玩家選擇轉成下一個視覺小說節點，包含劇情、選項與可能結局；規則系統仍負責數值邊界、存檔、顯示與 fallback。也就是說，遊戲比較像把 LLM 對話視覺化，但仍保留基本可控性。

## LLM API 模式的限制

- API Key 放在前端只適合玩家輸入自己的 key，不適合放站方私鑰。
- 不同 LLM 供應商的 CORS、endpoint 與模型名稱可能不同。
- API 呼叫會產生玩家自己的供應商費用。
- 若 API 失敗，系統會使用陽春版規則劇情。

因此目前採用「規則式生成為底，LLM API 增強文字」的混合架構：可以體驗生成式劇情，又不會讓 API 失敗時整個遊戲不能玩。
